1. CURRENT PROJECT STATE SUMMARY
============================================================

Project purpose:
NovelAI is a web-first Japanese novel platform for crawling/importing source material, queueing translation work, translating chapters through provider integrations, editing translated chapters, exporting books, and serving a public reader UI.

Current architecture shape:
- Backend: FastAPI backend under backend/src/novelai/, with API routers, services, activity/job runtime, source adapters, infrastructure HTTP/fetching, providers, prompts, storage, translation pipeline, scheduler, QA, and export logic.
- Frontend: Next.js frontend under frontend/, with public reader pages, admin pages, reusable components, frontend API client, frontend types, formatting helpers, and state utilities.
- Storage: local file-backed runtime storage under storage/novel_library/ by default. Storage is JSON-backed today and includes novel metadata, chapter bundles, translation versions, activity/job records, requests, usage logs, assets, exports, scheduler state, provider request records, chunk outputs, fetch cache, and traceability records.
- Docs: active project documentation is under docs/. The canonical active architecture document is docs/architecture/architecture.md. Supporting docs include readme.md, docs/guides/GETTING_STARTED.md, docs/reference/DATA_OUTPUT_STRUCTURE.md, docs/reference/PYTHON_COMMANDS.md, and docs/sources/codex_project_rules_guardrails.txt.

Current mode:
- Single-owner / controlled-admin.
- Scheduler-enabled admin-owned provider/model routing exists.
- Baseline owner/admin security hardening exists.
- Public user authentication is not implemented.
- Public contribution credentials are not implemented.
- Database storage is not implemented.
- Batch mode is not implemented.
- Billing, organizations, multi-admin teams, and production multi-tenant features are not implemented.

Structural health judgment:
- The project is not dead, but it is carrying planning/documentation scar tissue from rapid architecture evolution.
- Documentation appears more consolidated now than earlier, but the risk of old plans, audits, prompt summaries, and historical docs being treated as current truth remains high.
- The architecture is usable for future development if Codex consistently treats docs/architecture/architecture.md as canonical and refuses to mix phases.
- The codebase is likely maintainable, but only if cleanup happens in boring, staged passes. A heroic all-at-once refactor would be developer arson in a nice jacket.

Main risks:

- Runtime storage could be deleted, migrated, exposed, or renamed casually.
- Backend/frontend API contract drift could occur through inconsistent naming, legacy aliases, or partial schema updates.

2. CLEANUP STRATEGY
============================================================

The cleanup strategy is staged, not explosive.

Why docs should be cleaned first:
- The architecture document is the compass. If the compass is wrong or duplicated, every code change becomes guesswork.
- Current and historical docs may contain overlapping concepts from earlier prompts, audits, and frontend/backend plans.
- Codex needs a clear canonical source of truth before touching naming, API contracts, storage, frontend UI, or backend service boundaries.
- Docs cleanup is lower risk than source-code cleanup, but it has high leverage.

Why naming/API contracts should come before large refactors:
- Renaming after refactoring multiplies churn.
- API/storage/frontend naming inconsistencies can make valid refactors look broken.
- Canonical naming must be agreed before moving code across modules.
- Legacy aliases may be required for backward compatibility; removing them too early could break storage, tests, or frontend calls.

Low-risk changes:
- Updating documentation to clarify canonical architecture, current mode, blocked phases, and known debt.
- Creating planning files under docs/plans/.
- Archiving clearly historical prompt summaries or old audits only after they are proven superseded and the prompt explicitly permits archiving.
- Extracting small frontend utilities/components when duplication is obvious and tests/typecheck/build still pass.
- Thinning a backend router only when the extraction preserves behavior and has targeted tests.

High-risk changes:
- Storage schema migration.
- Runtime storage deletion or layout changes.
- Public contribution credential implementation.
- Auth/authorization model changes.
- Provider credential lifecycle changes.
- Scheduler pause/resume behavior changes.
- Chunk output/provider request traceability changes.
- Backend package flattening from backend/src/novelai to another layout.
- Broad API route renames.
- Broad frontend route rewrites.
- Database migration.
- Batch translation.

Changes to delay:
- Public contribution credentials until the readiness gate is Ready.
- Database support until there is a real scale/product reason and migration plan.
- Batch mode until synchronous translation, scheduler, provider errors, storage contracts, and QA are reliable.
- Backend package flattening until it is a dedicated mechanical migration.
- Adding new source sites until FetchService, source quality gates, and parser fixtures are stable.
- Large architecture rewrites until documentation, naming, and low-risk cleanup are finished.

Recommended principle:
Run one phase, inspect diff, run checks, commit, then move to the next phase. Do not stack phases like pancakes. Pancakes are good; mixed refactor diffs are not.
