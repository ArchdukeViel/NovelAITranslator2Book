# Design: Admin Audit Log Viewer

## Overview

`admin-audit-log-viewer` adds an admin-facing interface for reviewing audit events generated by sensitive system actions.

Earlier specs introduce audit logging for admin user management, notification preferences, public glossary annotation settings, export re-runs, backup/restore actions, and other operational controls. This spec adds the viewer layer: searchable, filterable, redacted, admin-only audit history with detail views.

This is an operator-readiness and security feature. It should help admins answer: who changed what, when, from where, and what safe metadata was recorded.

## Goals

* Add an admin audit log viewer page.
* Display audit events in reverse chronological order.
* Support filtering by actor, action, target type, target ID, status, date range, and severity.
* Support audit event detail view.
* Show before/after values where safe.
* Redact sensitive fields.
* Support pagination.
* Support request ID / correlation ID lookup where available.
* Add owner-only access controls (require_role("owner")).
* Add tests for authorization, filters, redaction, pagination, and detail rendering.

## Non-goals

* No new audit event generation for every feature.
* No full SIEM integration.
* No immutable append-only ledger redesign.
* No cryptographic signing of audit records.
* No public/user-facing activity feed.
* No real-time audit streaming.
* No arbitrary raw database browsing.
* No exposure of secrets, tokens, prompts, raw text, or private content.
* No multi-role scoped permission model (like audit:read).

## Status values

Recommended statuses:

```text id="sgnupz"
succeeded
failed
denied
partial
unknown
```

Meanings:

```text id="hp9uif"
succeeded: action completed
failed: action attempted but failed
denied: authorization/policy blocked action
partial: some parts completed and some failed
unknown: status missing or legacy event
```

## Severity values

Recommended values:

```text id="yfzpqy"
info
warning
critical
```

Examples:

```text id="bwsgw6"
info: normal admin read/update
warning: user disabled, setting changed, export re-run
critical: role changed, sessions revoked, backup restore started, security setting changed
```

## Important audit actions

```text id="662vh7"

```

Do not require every action to have a custom UI. Unknown action names should still render safely.

Filter behavior:

```text id="kdtuc7"
filters update query params where frontend convention supports it
clear filters resets to default view
invalid date range shows validation error
large date ranges may be capped
```

## Search

Optional search can support:

```text id="cpo60l"
action contains
target label contains
actor display name contains
request ID exact match
correlation ID exact match
```

Avoid free-text search over raw metadata if it risks exposing or relying on sensitive content.

## Redaction

Audit viewer must redact sensitive fields before display.

Never display:

```text id="qg39e3"
passwords
password reset tokens
API keys
provider keys
auth tokens
session tokens
signed URLs
raw prompts
raw source chapter text
raw translated chapter text
private glossary definitions
full request bodies
full IP addresses
storage credentials
private filesystem paths
```

Recommended redaction marker:

```text id="luy61h"
[redacted]
```

Redaction should happen server-side. Frontend may also defensively hide known unsafe keys.

## Before/after changes

Changes should be shown only for safe fields.

Recommended display:

```text id="to9n50"
field
before
after
```

Examples:

```text id="0qlsjy"
role: user -> admin
status: active -> disabled
public_glossary_annotations_mode: inherit -> disabled
```

For unsafe fields:

```text id="ec6rz8"
password: [redacted] -> [redacted]
api_key: [redacted] -> [redacted]
```

If change diff is unavailable:

```text id="4hxzpf"
No field-level changes recorded.
```

## Authorization

Admin-only.

Rules:

```text id="2b0zvc"
unauthenticated -> 401 or login redirect
authenticated non-admin -> 403
admin -> allowed
disabled admin -> blocked according to existing auth policy
```

If scoped permissions exist:

```text id="o4a59n"
audit:read
audit:export optional
```

## Exporting audit logs

Optional CSV/JSON export can be useful, but it is sensitive.

Recommended V1:

```text id="5q0z4l"
No export unless already required.
```

If export is implemented:

```text id="8gxtsu"
admin-only
same filters as viewer
server-side redaction
max row limit
audit event generated for export
no secrets/raw private data
```

## Retention

Audit retention should follow security policy.

Recommended config:

```text id="jkdyfi"
AUDIT_LOG_RETENTION_DAYS=365
AUDIT_LOG_VIEWER_MAX_DATE_RANGE_DAYS=90
AUDIT_LOG_PAGE_SIZE_DEFAULT=50
AUDIT_LOG_PAGE_SIZE_MAX=200
```

Audit cleanup should not be implemented here unless maintenance cron already supports it. If cleanup exists, viewer should handle old missing data gracefully.

## Error handling

Expected UI states:

```text id="3wsgix"
loading audit events
empty audit events
filtered empty
load error
detail load error
forbidden
unauthorized
invalid filters
```

Safe messages only. Do not show raw backend stack traces.

## Observability

Viewing audit logs is itself sensitive.

Recommended audit events:

```text id="zaqnyd"
admin.audit.viewed
admin.audit.detail_viewed
admin.audit.export_requested
```

To avoid noisy logs, list-view auditing can be optional or sampled. Detail views and exports should be audited.

Safe fields:

```text id="4l75la"
admin_user_id
event_id
filters_summary
created_at
```

Do not audit raw metadata payloads.

## Accessibility

Audit viewer should be accessible:

```text id="alhidx"
table has headers
filters have labels
status/severity are not color-only
detail drawer has focus management
pagination is keyboard accessible
error/empty states are readable text
```

## Rollout plan

1. Inspect audit event storage and existing admin audit API.
2. Add missing list/detail API fields.
3. Add server-side redaction.
4. Add admin audit page.
5. Add filters and pagination.
6. Add detail view.
7. Add before/after diff display.
8. Add safe empty/error states.
9. Add optional audit-of-audit viewing.
10. Add tests.
11. Verify:

    * admins can review sensitive changes.
    * non-admins cannot access logs.
    * unsafe fields are redacted.
    * filters and details work.
