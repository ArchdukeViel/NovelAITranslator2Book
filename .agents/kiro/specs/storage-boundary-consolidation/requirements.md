# Requirements: Storage Boundary Consolidation

## Introduction

Technical audit 5 flags hybrid storage as critical debt. Current canonical architecture intentionally uses Postgres-backed metadata plus private file-backed chapter content under `storage/novel_library`. This spec does not force all content into one database. It consolidates access behind backend storage/database boundaries, removes ad hoc path coupling, and prevents API/frontend exposure of runtime paths.

## Requirements

### REQ-1: Canonical Storage Boundary

All novel content and metadata access must go through approved backend boundaries.

- REQ-1.1: Metadata, users, jobs, requests, audit, and settings must use `novelai.db` repository/session patterns.
- REQ-1.2: Heavy chapter content must use `novelai.storage` services or repositories, not direct arbitrary filesystem access from services, routers, or frontend code.
- REQ-1.3: API responses must return IDs, keys, or public URLs generated by backend services; never raw filesystem paths.
- REQ-1.4: `storage/novel_library` must remain private and must not be mounted as static public files.

### REQ-2: Legacy Path Inventory and Deprecation

Legacy raw/translated path usage must be known and reduced safely.

- REQ-2.1: Add an inventory of direct runtime-storage path reads/writes in backend code.
- REQ-2.2: Replace direct path usage in API routers with service/storage calls.
- REQ-2.3: Replace direct path usage in orchestration code only where behavior can be preserved.
- REQ-2.4: Raw scraped chapter files must not be silently deleted after translation.

### REQ-3: Migration Safety

Existing local novels must remain readable.

- REQ-3.1: Any migration or compatibility loader must preserve existing `novel_id` and `chapter_id` values.
- REQ-3.2: Migration code must validate before writing destructive changes.
- REQ-3.3: Destructive cleanup must require explicit owner action, not automatic startup behavior.

### REQ-4: Tests and Contract Checks

Storage behavior must be covered by regression tests.

- REQ-4.1: Tests must cover reading legacy chapter content through canonical storage services.
- REQ-4.2: Tests must verify public/admin APIs do not expose raw filesystem paths.
- REQ-4.3: Tests must verify missing/corrupt content returns safe error envelopes, not raw tracebacks.

## Non-Goals

- No full rewrite to database-only chapter content.
- No object storage provider implementation.
- No deletion of existing raw scraped content.
- No frontend direct storage access.
